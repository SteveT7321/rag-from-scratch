# 05 — Agentic RAG（ReAct）

LLM 自主決定何時搜尋、搜尋什麼關鍵字、何時停止並回答。
不再是固定的「查詢→搜尋→生成」管線，而是動態的推理-行動迴圈（ReAct）。
索引共用 `01_naive_rag/index/`，不需要重新 ingest。

---

## 原理：ReAct 迴圈

ReAct = **Re**asoning + **Act**ing。每一步 LLM 輸出三行：

```
思考：[分析目前狀況，決定下一步]
工具：search | lookup | finish
輸入：[工具的輸入參數]
```

三種工具：

| 工具 | 功能 |
|------|------|
| `search(查詢)` | 向量搜尋，回傳 Top-3 相關片段 |
| `lookup(頁碼)` | 取得指定頁的完整文字內容 |
| `finish(答案)` | 輸出最終答案，結束迴圈 |

最多執行 6 步；若達到上限仍未 finish，用最近 3 筆觀察強制生成答案。

---

## 架構流程

```
問題
 │
 ▼ Step 1：思考 → 決定用哪個工具
 │
 ▼ 執行工具 → 取得觀察結果（Observation）
 │
 ▼ Step 2：思考（加入觀察）→ 再決定
 │
 ▼ ...（最多 6 步）
 │
 ▼ finish(答案) → 輸出
```

每一步的「歷史記錄」都附在 Prompt 裡，LLM 能看到所有先前的搜尋結果再做決策。

---

## 使用方式

```bash
# 一般模式（只顯示每步的「思考」）
python agentic_rag.py chat -q "BRZ 的引擎馬力是多少？"

# Verbose 模式（顯示完整推理過程）
python agentic_rag.py chat -q "BRZ 的引擎馬力是多少？" --verbose

# 互動模式
python agentic_rag.py chat --verbose
```

## 實測結果

索引：25 chunks（共用 01_naive_rag/index/）

### 查詢：「BRZ 的引擎馬力是多少？」（--verbose）

共執行 **6 步**才找到答案：

| 步驟 | 工具 | 輸入 | 思考摘要 |
|------|------|------|---------|
| 1 | search | BRZ 馬力 | 需要搜尋馬力數值 |
| 2 | search | BRZ 規格 | 結果只有描述性文字，沒有數值 |
| 3 | search | SUBARU BRZ 規格 | 結果仍是行銷描述，嘗試換措辭 |
| 4 | search | 動力規格 | 繼續找技術參數頁 |
| 5 | search | BRZ 動力規格 馬力 | 更精準的關鍵字組合 |
| 6 | finish | — | 第 5 步終於在第 2 頁找到「234 / 7,000」 |

最終回答：**Subaru BRZ 搭載 2.4L 水平對臥四缸自然進氣引擎，馬力 234 hp。**

> 觀察：Agent 使用了所有 6 步配額，每步都在思考「為何沒找到數值」並調整關鍵字策略。
> 這展現了 ReAct 的自適應特性，但也顯示出 25 個 chunk 的索引稀疏，搜尋較難精確命中。

## 效果分析

### 優點
- **自適應搜尋**：根據觀察結果動態調整關鍵字，不依賴單一 embedding 命中
- **透明推理**：`--verbose` 完整呈現每步思考過程，可解釋性高
- **可擴充工具**：只需新增函式即可加入計算、外部 API 等工具
- **多跳推理**：可以先 search 找頁碼，再 lookup 取完整頁面

### 缺點
- **延遲高**：每步都要呼叫一次 LLM，6 步 = 6 次 generate，遠比單次管線慢
- **格式脆弱**：LLM 必須嚴格輸出「思考：/工具：/輸入：」格式，gemma4 偶爾格式不對需容錯處理
- **步數限制**：MAX_STEPS=6 是硬上限，複雜問題可能不夠
- **索引稀疏時效率低**：若正確 chunk 很難命中，可能用完所有步數仍未找到

---

## 關鍵設定

| 參數 | 值 | 說明 |
|------|----|------|
| MAX_STEPS | 6 | 最大推理步數 |
| SEARCH_TOP_K | 3 | 每次 search 回傳幾個片段 |
| GEN_MODEL | gemma4:e4b | 推理與生成共用同一模型 |

---

## 與其他方案比較

| 方案 | 搜尋策略 | 靈活性 | 延遲 |
|------|---------|-------|------|
| 01 Naive | 固定單次查詢 | 低 | 最快 |
| 06 RAG Fusion | 固定多查詢 + RRF | 中 | 中等 |
| **05 Agentic** | **LLM 自主決策** | **高** | **最慢** |

適合用在：問題複雜、需要多步推理、或使用者問題措辭難以預期的場景。
